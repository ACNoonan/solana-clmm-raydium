## 3. Method/structure narrative

**Naming and signatures.** The crate inherits Raydium's naming wholesale: `get_sqrt_price_at_tick` (vs Uniswap's `get_sqrt_ratio_at_tick`), `get_tick_at_sqrt_price` (vs `get_tick_at_sqrt_ratio`), Q64.64 rather than Q96, fees in PPM rather than hundredths-of-a-bip with the Solidity `uint24` quirk. This is the right call — the README explicitly markets the crate as the byte-for-byte port — but it does mean a downstream multi-AMM router has to write its own adapter layer to talk to both. Orca's `tick_index_to_sqrt_price` and Uniswap's `get_sqrt_ratio_at_tick` are also mutually incompatible names, so there's no good winner here. The crate's choice of mirroring upstream Raydium is defensible; the cost is one more transliteration in calling code.

**Argument ordering.** `compute_swap_step(sqrt_price_current, sqrt_price_target, liquidity, amount_remaining, fee_rate, is_base_input, zero_for_one, block_timestamp)` mirrors upstream Raydium exactly. Compare to Uniswap's `compute_swap_step(price, price_target, liquidity, amount_remaining: I256, fee)` — there `is_base_input` is encoded as the *sign* of `amount_remaining` (i.e. `I256` doubles as direction flag). Orca's internal version takes 7 explicit args. The new crate is fine here; the only smell is the unused `block_timestamp` arg, which exists upstream because Raydium's *test* version branches on it but the production `#[cfg(not(test))]` branch does not. Our crate copied the production behaviour, so the parameter is dead weight. **For v0.2 you can drop it** without breaking anything; flag it `#[deprecated]` first.

**Error handling.** Three philosophies in play. Uniswap V3 → `thiserror` enum, `Result<T, UniswapV3MathError>`, error variants name the Solidity revert string. Orca → const `&str` `CoreError`s. Ours → C-enum `ErrorCode` with no `Display`/`Error` impl, no `thiserror`. That is a regression vs both peers: `solana-clmm-raydium`'s `ErrorCode` is `Debug + Clone + Copy + PartialEq + Eq` only. Consumers can't `?`-bubble through `anyhow::Error` without adapting, and there's no human-readable reason string in panics. A 10-line `impl Display` + `impl std::error::Error` (or `core::error::Error` on stable Rust 1.81+) would close the gap with no semantic change.

**Module layout.** Good instinct: 4 implementation modules are `#[doc(hidden)] pub mod` (`big_num`, `fixed_point_64`, `full_math`, `unsafe_math`), 7 are public, with a curated re-export block at the lib root. This is the best layout I've seen in the comparison set — `uniswap_v3_math` re-exports nothing at the crate root, you have to know that `compute_swap_step` lives in `swap_math::compute_swap_step`. Orca re-exports everything flat. Ours strikes a real balance.

However, the curated re-export is *under-curated* — `get_next_sqrt_price_from_input/output`, `get_delta_amount_0/1_signed`, and the bitmap helpers (`most_significant_bit`, `check_current_tick_array_is_initialized`) are reachable as `solana_clmm_raydium::sqrt_price_math::get_next_sqrt_price_from_input` but absent from the lib-root re-exports. If they're public API, hoist them; if not, demote them to `pub(crate)`. The current state is "pub by accident". Uniswap's surface contains every primitive at `uniswap_v3_math::sqrt_price_math::*`, intentionally; that's a stronger contract.

**Big-int exposure.** `U128/U256/U512/U1024` are all `#[doc(hidden)]`. That's correct for an arithmetic library — you don't want consumers building positions against your internal types. But `next_initialized_tick_array_start_index` takes `bit_map: U1024`, which is a hidden type in the public signature. Either expose `U1024` (uniswap exposes `alloy_primitives::U256` everywhere), or accept `[u64; 16]` and convert internally. Today, anyone calling that function has to import `solana_clmm_raydium::big_num::U1024` from a doc-hidden module, defeating the doc-hidden marker.

**`no_std`.** Cargo.toml lists `categories = ["finance", "mathematics", "no-std"]` — but `liquidity_math.rs` uses `std::mem::swap` six times. **(Fixed in v0.1.0 — replaced with `core::mem::swap`.)**

**Feature flags.** Zero. Worth adding: `serde` (for fixture/state interop), `wasm` (for an in-browser quoter — orca proves this is a real use case). Not urgent for v0.2 but cheap.

**v0.2 priority list (from peer comparison, most-to-least impact):**
1. `compute_swap_full` — every peer that ships orchestration ships this, and it is the function most LVR/MEV consumers actually want. The `tests/replay.rs::replay_swap` already implements 95% of this; promote it from a private test helper to a public function, fix the `Result` type, gate behind a `swap` feature flag à la orca, and ship.
2. Implement `Display`/`Error` on `ErrorCode`.
3. Public `cross()` helper that wraps `add_delta` with the zero-for-one negation logic. Right now that logic lives **only** in `tests/replay.rs:200` and every consumer reimplements it.
4. Hoist or demote the partially-public functions to fix the curated re-export.
5. Restore upstream proptests with full domain coverage.

## 4. Test suite audit — verdicts per test

The bar: **does this test fail when the implementation is broken in a relevant way?** Verdicts are **load-bearing** / **partially load-bearing** / **sham**. Sham can mean tautological, skip-on-empty, type-checks-only, or so narrow it never tests an edge.

### 4.1 `tests/smoke.rs` — 3 tests

- `boundary_constants_round_trip` — **load-bearing**. Asserts `get_sqrt_price_at_tick(MIN_TICK) == MIN_SQRT_PRICE_X64` and same for MAX. These are the two hardest values to get right; they verify the magic-table rounding products at the boundary. If the tick table is off by one bit anywhere in the high-order corrections, this fails.
- `tick_zero_is_q64` — **load-bearing**. The neutral-element test: `tick=0 → sqrt_price = 2^64`. If the early `ratio = U128([0, 1])` is broken, this catches it instantly.
- `tick_round_trip_sample` — **load-bearing**. 11 manual ticks including signed boundaries.

### 4.2 `tests/properties.rs` — 8 tests (5 plain + 3 proptest)

- `tick_round_trip_sampled` — **load-bearing**. Steps 100, ~8.8k samples. Hit by any broken correction factor.
- `tick_round_trip_exhaustive` (`#[ignore]`) — **load-bearing but CI-invisible**. Exhaustive 887k iterations is the gold standard, but `#[ignore]` means it does not run unless `--ignored` is passed. **CI gap**: add a release-mode CI job that runs `cargo test -- --ignored` on this one. Otherwise the strongest test in the suite never executes.
- `max_boundary_is_one_way` — **load-bearing**. Verifies the `< MAX_SQRT_PRICE_X64` half-open interval; this is exactly the kind of off-by-one that broke older Solana CLMM ports.
- `sqrt_price_is_monotonic_in_tick` — **load-bearing**. Fundamental invariant.
- `out_of_domain_ticks_error` — **load-bearing**. 4 boundary error cases.
- `proptest_round_trip` — **load-bearing**. 256 cases over the full tick domain. Will catch any silent bug in the `tick_low/tick_high` selection in `get_tick_at_sqrt_price`.
- `swap_step_zero_for_one_price_decreases` — **partially load-bearing**. Asserts price-monotonicity, target-not-overshoot, `amount_in <= amount_remaining`. This is good for monotonicity but *does not assert any of*: amount_out monotonicity in liquidity, fee-amount magnitude, that fee_amount + amount_in actually equals the right value, that `add_delta` would round-trip with the resulting state. The range `current_tick in -100_000..100_000` and `delta in 1..5_000` is **far** from boundary; you never test against `MIN_TICK`, never test `liquidity = 0` (would panic — see assertion in `sqrt_price_math.rs:115`), never test `fee_rate = 0` or `fee_rate = FEE_RATE_DENOMINATOR_VALUE - 1`. Upstream Raydium's removed proptest used `MIN_SQRT_PRICE_X64..MAX_SQRT_PRICE_X64` for both prices and `1..u32::MAX as u128` for liquidity — far wider. The README says upstream proptests were "removed in favor of new tests anchored to mainnet ground-truth"; in practice the new proptest is **a strict subset** of the removed one. That's a quiet regression.
- `swap_step_one_for_zero_price_increases` — **partially load-bearing**. Same critique. Same too-narrow ranges.
- `swap_step_fee_is_bounded` — **partially load-bearing**. Asserts `amount_in + fee <= amount_remaining`. Fine, but the upstream proptest also asserted `(amount_in + fee == amount_remaining) when sqrt_price_next != target` — a tighter equality, not just a bound. Worth adding.

### 4.3 `tests/decoders.rs` — 2 tests

- `pool_state_decodes_from_fixture` — **sham at CI level, load-bearing locally**. The skip-on-missing-fixture pattern means a fresh checkout with no fixtures (a junior PR contributor running `cargo test`, or a CI job that cloned without LFS, or a future fork that deletes fixtures) prints "[skip]" and the test passes. There is no `cargo test --` flag that fails on skip-with-message. The test has good content (asserts plausible tick range, asserts `tick_spacing == 1`, asserts the round-trip `tick_from_sp == pool.tick_current`) but **none of that runs without the fixture file**. The fix is one of: (a) `panic!` instead of `return` on missing fixture, (b) gate the test behind `#[cfg(feature = "fixtures")]` so absent fixtures = absent test instead of passing test, (c) commit a tiny synthetic fixture for the smoke path. Option (c) is best because the fixtures are ~50KB each and already committed (17 are present in this checkout) — but since you can't tell from the source whether they'll be there in someone else's checkout, the test should fail fast.
- `tick_array_decodes_from_fixture` — same pattern, same verdict. Same fix.

### 4.4 `tests/replay.rs` — 1 test (the headline test)

- `replay_fixtures_match_observed` — **load-bearing when fixtures exist; sham at the CI level**. Same skip-on-empty pattern. When fixtures are present, this is a genuine differential test against mainnet ground truth — it walks the tick array, calls `compute_swap_step` repeatedly, applies `add_delta` on tick crosses, and asserts byte-exact match against `observed_amount_in` and `observed_amount_out`. That is **the strongest test in the suite by a wide margin**; the fixtures are real Helius-pulled mainnet snapshots with the swap result captured from chain.
  - **However**: the `observed_amount_in` / `observed_amount_out` come from `swap["observed_amount_in"]` in the fixture JSON. If `scripts/fetch_fixtures.py` itself has a bug — say, it parses the wrong inner-instruction's amounts — every fixture is consistent and every replay matches. The fixture-generation script is in Python and is not tested. Mitigation: at least one fixture should be hand-verified against a Solana explorer.
  - The walking logic deviates subtly from the on-chain swap: ours flattens **all** initialized ticks across all snapshotted arrays into a single sorted Vec and uses linear scan, whereas on-chain Raydium uses the tick-array bitmap to find the next array, then `TickArrayState::first_initialized_tick` / `next_initialized_tick` within that array. **Functionally equivalent on the test fixtures because the on-chain swap only touched arrays that the snapshot includes**, but it does not exercise `next_initialized_tick_array_start_index` — your only public bitmap function. Replay test passes, bitmap function is never called. That is a real coverage gap.
  - Hardcoded `steps < 256` cap silently terminates a runaway swap without failing the test. If there's a bug that puts the loop into a no-op state where neither `amount_remaining` nor `sp_current` advances, the test will eventually error on the **next** assertion (the `pred_in != obs_in` check), but you lose the proximate cause. Replace with `panic!("swap exceeded 256 steps — likely infinite loop")`.

### 4.6 Property invariants not asserted

- **Liquidity conservation across tick crosses.** When a swap crosses a tick, the pre-cross and post-cross liquidity must differ by exactly `liquidity_net` (negated for zero_for_one). Replay tests this implicitly but never as a standalone invariant.
- **Swap-output monotonicity in liquidity.** Holding all else constant, `compute_swap_step` should produce greater `amount_out` for greater `liquidity`.
- **Swap symmetry / round-trip.** A swap `0→1` followed by `1→0` for the resulting amount should not exceed the original input.
- **Liquidity↔amounts round-trip.** `get_liquidity_from_amounts(sp, sp_a, sp_b, amt0, amt1)` then `get_delta_amounts_signed(...)` should give back `(amt0, amt1)` up to rounding. Currently zero coverage of either direction.
- **`get_sqrt_price_at_tick(t+1) > get_sqrt_price_at_tick(t)` strict everywhere.** You assert this with step 100 in `sqrt_price_is_monotonic_in_tick`. Upstream's removed proptest asserted it for **every adjacent pair** via `get_sqrt_price_at_tick_test (tick in MIN_TICK+1..MAX_TICK-1)`. Worth restoring.

### 4.7 Edge cases the proptests don't reach

- `current_tick = MIN_TICK + 1`, `target_tick = MIN_TICK` (boundary swap)
- `current_tick = MAX_TICK - 1`, `target_tick = MAX_TICK` (other boundary)
- `liquidity = 1` and `liquidity = u128::MAX`
- `fee_rate = 0` (no-fee pool — Raydium has these for OEV-protected pools)
- `fee_rate = 999_000` (the upper edge — pools can theoretically configure up to ~99.9%)
- `amount_remaining = 1` (dust input)
- `amount_remaining = u64::MAX` (max input — overflow trigger)
- `delta = 1` (sub-tick price moves)

The current proptest range `current_tick in -100_000..100_000`, `liquidity in 1_000..1_000_000_000_000` is conservative SOL/USDC-ish state space; it never tests boundaries.

### 4.8 Differential testing — what litesvm would buy you

Mainnet replay tests against the *same* compiled program that ran on chain — so any drift between extracted source and deployed bytecode (Raydium upgrades the program, you don't update extraction) is invisible to your tests until a fixture starts failing. A `litesvm` differential test would:
1. Load the actual on-chain program ELF.
2. For each property test case, run both the in-memory extracted math and the program in `litesvm`, assert byte-exact match.
3. Catch any future upstream change immediately in CI without needing to capture new mainnet fixtures.

Meteora's `commons/tests/integration/test_swap.rs` does exactly this and it's the gold standard for AMM math correctness. The CHANGELOG says "Litesvm differential test" is a v0.2 deferral; that is the right choice.
